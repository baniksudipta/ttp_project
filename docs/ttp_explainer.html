Travelling Thief Problem — Visual Explainer

A quick refresher on what you built and why it was hard

1. What is the Travelling Thief Problem?

🗺️ TSP Part

Visit all cities exactly once and return home. Find the shortest route.

+

🎒 Knapsack Part

At each city, items are available. Pick items to maximize profit without exceeding weight limit.

≠

⚡ The Twist

More items = heavier bag = slower travel = more rent paid. The two problems fight each other.

Objective = Profit collected − (Time taken × Rent rate)

A great route but no items = zero profit. Grabbing everything = too slow, too much rent. You need a balance.

2. How a Solution ("Thief") is Represented

🗺️ Tour (theif.tour)

An ordered list of city indices, e.g.:

[3 → 1 → 4 → 2 → 5 → 3]

Stores total distance cost and a distance-remaining array for each city.

+

🎒 Sack (theif.sack)

A binary vector — one bit per item:

[1, 0, 1, 1, 0, 0, 1, ...]

1 = steal this item, 0 = leave it. Stores total weight and total cost.

=

🧑‍💼 Theif Object

One candidate solution. Has a profit score computed by simulating the full tour with items collected.

Higher profit = better solution.

3. Artificial Bee Colony — The Main Algorithm

16 "bees" each carry one solution (thief). They cooperate to find a good solution over 40 iterations.

🐝 Employed Bees (16)

Each owns a solution. In every iteration, tries to improve it via local search. If no improvement for too long → becomes a scout.

👀 Onlooker Bees (8)

Watch employed bees. Preferentially pick the better-looking solutions (roulette wheel) and also try to improve them.

🔍 Scout Bees

If a bee is stuck (no improvement after N tries), it's reset with a completely random new solution. Escape local optima.

↻ ITERATION LOOP (×40)
1

Employed bees update: every bee runs local search on its solution (2-opt + Tabu Search, random order)

2

Onlookers sent: 8 bees pick good solutions by roulette wheel and run local search on them too

3

Scouts sent: any bee stuck too long gets randomly reinitialised

4

Track best: record the highest-profit solution seen so far

4. Local Search — How Each Bee Improves Its Solution

Two methods, applied in random order each time:

2-opt (fixes the Tour)

  • Look for two route edges that cross each other
  • If un-crossing them improves profit → swap
  • Repeat until no crossing edges improve things

Tabu Search (fixes the Sack)

  • Generate neighbours by flipping bits in the knapsack vector (take/leave item)
  • Pick the best neighbour that's not in the tabu list (recent memory)
  • Add it to tabu list → prevents revisiting same states
current: [1,0,1,1,0]
neighbour:[1,0,0,1,0] ← flip bit 3
→ if better & not tabu: accept

5. Knapsack Repair — Keeping the Bag Valid

After any change, the bag might be overweight. modify_sack.m fixes it:

80% of the time: Greedy

Remove items with the worst weight-to-profit ratio first — heavy but cheap items go first.

20% of the time: Random

Remove a random item. Adds exploration — avoids always converging to the same knapsack.

There's also a smart scoring trick: items near the end of the tour are ranked lower (you carry them a short distance anyway), so they're removed first.

6. Why Results Weren't Great — The Core Tension

Better route
(shorter tour)

⟺

More items
(heavier bag)

Improving the tour doesn't mean profit goes up — a shorter route with fewer items can be worse than a longer route with many items. And vice versa. The two parts of the solution are entangled.

What the code does

  • 2-opt improves tour independently
  • Tabu Search improves knapsack independently
  • No step jointly optimizes both

The limitation

  • 16 bees × 40 iterations = limited exploration
  • Local search can get stuck in local optima
  • The real optimum requires co-evolving tour + sack together
